# ADR 0009: Triage to GitHub Issues — gh-CLI Client, Body-Marker Idempotency, Dry-Run Default

## Status

accepted

## Date

2026-06-02

## Context

Phase 14 (`rag-triage`) produces a deterministic `results/triage.json` — failure clusters
keyed on `(failure_mode, category)`, a `dominant_cluster` pointer, and a representative
example per cluster. That artifact had no consumer: acting on the dominant failure cluster
meant hand-authoring a GitHub Issue and transcribing the stats by hand.

Phase 15 (`rag-issues`) closes the loop by drafting one Issue per failure cluster, grounded
in the real cluster stats plus a concrete representative example. Three decisions needed
recording, because creating a GitHub Issue is a hard-to-reverse, outward-facing side effect:

1. **GitHub integration boundary** — how to talk to GitHub (CLI vs library/REST).
2. **Idempotency** — how a re-run avoids spamming duplicate Issues, given GitHub has no
   server-side uniqueness on Issues.
3. **Side-effect safety** — how the default path stays safe while still enabling real
   creation.

A pre-design research scan (Exa) confirmed the production-dominant idempotency pattern is a
deterministic marker embedded in the issue body plus a pre-create search for that marker
(seen in `github/gh-aw`, `gastownhall/beads`, `octokit-plugin-unique-issue`).

## Decision

**1. `gh` CLI as the default client, behind a `GitHubClient` Protocol seam.**
A two-method `GitHubClient` Protocol (`search_issues`, `create_issue`) lives in
`eval/github.py`. The default `GhCliClient` shells out to the `gh` CLI via
`subprocess.run([...])` with an **argument list — never a shell string** (the markdown body
and fingerprint are passed verbatim, with no quoting/injection hazard). `gh` is chosen over
PyGithub/REST for v1: zero new runtime dependencies, ambient auth (`gh auth` / `GH_TOKEN`),
and indexed `gh issue list --search "<fingerprint> in:body"`. The Protocol is the localized
swap point should we ever move to PyGithub/REST — that swap is a single-file change.

**2. Body-marker fingerprint idempotency, keyed on `(failure_mode, category, schema_version)`.**
Each draft embeds a hidden HTML comment
`<!-- rag-triage-cluster: {failure_mode}|{category} schema={schema_version} -->`. The
search fingerprint (`rag-triage-cluster: {failure_mode}|{category} schema={schema_version}`)
is a substring of that marker, so a body search finds it. The marker and the fingerprint are
generated by a single shared `_cluster_identity` helper, so they cannot drift. The
`schema_version` is part of the key on purpose: a future triage schema bump (`"2.0"`)
deliberately produces a distinct fingerprint and re-files, rather than falsely deduping
against a v1 Issue.

**3. Dry-run / draft by default; `--create` is explicit opt-in; skip-on-existing.**
By default `rag-issues` writes one markdown draft per selected cluster to
`results/issues/<failure_mode>-<category>.md` (atomic temp-file + `os.replace`) and prints a
summary — **no network call occurs**. Only `--create` constructs a client and, per draft,
searches for the fingerprint and **skips with a message** if a matching open Issue exists,
otherwise creates it. Default selection is the dominant cluster only; `--all-clusters`
widens. The pure core (`eval/issues.py`) builds drafts with zero I/O, so it is fully
offline-testable; the create path is exercised in tests only through an injected fake.

## Consequences

- **Grounded, not a gadget.** Every draft is built from real `TriageCluster` numbers; there
  is no code path that emits an Issue not backed by a cluster in `triage.json`.
- **Safe by default.** No GitHub mutation is possible without `--create`; re-runs are
  idempotent; a failed draft write leaves no partial file.
- **Best-effort dedup (known limitation).** GitHub has no server-side Issue uniqueness, so
  the body-marker + pre-create search is best-effort, bounded by search-index propagation
  delay. Behavior degrades to "may create a duplicate if the index has not yet propagated,"
  never to a crash. Concurrent creators are out of scope.
- **Offline tests.** The `GitHubClient` seam is injected as a `FakeGitHubClient`; the `gh`
  impl is never instantiated in the suite, honoring the cassette/replay ethos (ADR-0006).
- **Forward-compat seam.** Moving to PyGithub/REST, or adding an LLM `DraftStrategy` for the
  body, is a localized change behind the existing seam — no rewrite.
- **`gh` is a runtime dependency only for `--create`.** Dry-run (the default) needs neither
  `gh` nor network.
